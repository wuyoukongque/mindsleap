场部门访谈形成的启动培训：不是通用 AI 课，而是围绕真实业务痛点的转型第一步。

核心判断 · 从工具使用走向组织工作流

AI 转型不是让每个人“多会几个 Prompt”，而是把嘻游记的高频经营动作，变成 可复用、可验收、可沉淀 的工作流。

今天的目标不是讲完一套概念，而是让每个部门开始拥有自己的第一个 Agent Demo。

访谈洞察 · 组织经验需要被流程化

九场访谈指向同一个问题：经验很多，但工作流还没有被系统化瓶颈不在努力程度，而在能否稳定复现、持续沉淀。

01

数据分散，决策滞后

美团、点评、抖音、收银、供应链、财务表格各自为政，很多复盘按月发生，错过了经营调整窗口。

02

流程靠人，标准难留存

研发、巡店、外卖、课件、对账、招商内容都有经验方法，但还没有形成可交给新人和 Agent 执行的规格。

03

资产在个人手里

Prompt、Skill、模板、调研报告、历史数据如果不进入公司级资产库，AI 转型很容易变成个人技巧。

01
第一部分 · 共同语言

从 Prompt 到餐饮经营工作的 AI 操作系统

先让管理团队、部门负责人和一线骨干拥有同一套 Agent 语言：任务、上下文、工具、Skill、验收与复用。

认知转变 · 不是会问，而是会编排

从“问 AI 一个问题”，到“让 AI 跑一条流程”真正的升级不是语言技巧，而是把工作拆成可执行、可验收的流程。

Prompt 是一次对话

适合临时灵感、写一段话、整理一个想法。问题是不可复用、不可审计、结果依赖个人表达。

Input -> AI -> Output

01

Workflow 是一套生产线

输入资料、处理步骤、判断标准、输出格式都写清楚。

02

Skill 是给 Agent 的 SOP

把经验变成文件，让新人和 AI 都能照着做。

03

Spec 是人与 AI 的工作合同

说明任务目标、边界、数据源、验收标准，减少返工。

Agent 结构 · 工具、记忆、指令与人工验收

Agent 不是万能助手，而是一组可编排的执行角色把目标、上下文、工具和验收标准组合起来，才能形成稳定产出。

1

任务目标

要解决什么业务问题，输出给谁使用。

2

上下文

品牌资料、历史数据、业务规则、参考案例。

3

工具

浏览器、表格、代码、文件、网页、数据接口。

4

验收

由人判断是否准确、可用、能进入日常流程。

从“我让 AI 帮我干点活”，升级为“我设计一条流程，让 Agent 按规则持续产出”。

Agent 制作 · 五层结构

做 Agent 不是写一个神奇提示词，而是搭一套执行系统一个可复用 Agent 至少要有角色、知识、流程、工具和验收五层。

01

角色定位

它服务哪个部门、解决哪个业务问题、输出给谁使用。角色越具体，结果越稳定。

02

知识资产

品牌资料、业务规则、历史报表、菜单、SOP、竞品样例，都是 Agent 的“教材”。

03

工作流程

把任务拆成步骤：读取、清洗、判断、生成、复核、输出，而不是一次性让它“给方案”。

04

工具能力

文件、表格、网页、浏览器、代码、数据接口，每个工具都要有明确使用边界。

05

验收闭环

谁来确认、看哪些指标、失败如何重跑、如何进入周会和公司资料库。

Agent 制作 · 从需求到交付

一条部门 Agent 的开发流水线：先 Spec，再 Demo，再 Skill不要跳过规格设计；跳过规格，后面一定靠反复返工补回来。

Step 1

选高频任务

优先选择每周重复、规则清楚、数据可拿到、部门负责人愿意验收的任务。

Step 2

写 Spec

写清业务目标、输入资料、处理步骤、判断规则、输出格式和验收标准。

Step 3

喂上下文

把真实表格、过往案例、品牌规范、失败样例和老板认可的输出一起给 Agent。

Step 4

做 Demo

先做网页、清单、表格、报告或脚本的小闭环，现场能打开、能解释、能修改。

Step 5

固化 Skill

把做法写成 Agent 可读取的 SOP：触发条件、步骤、规则、输出和注意事项。

Step 6

周度复盘

进入部门周会，用真实结果修正规则，决定继续试点、扩展或暂停。

Agent 制作 · Skill 是给 Agent 的 SOP

一个好的 Skill，把人的经验变成 Agent 可以照着执行的作业标准书Skill 不只是 Prompt，而是触发条件、流程、素材和验收标准的集合。

SKILL.md ├─ 什么时候使用 ├─ 输入资料要求 ├─ 执行步骤 ├─ 输出格式 ├─ 质量检查 └─ 常见失败处理

触发条件

例如“每周一生成外卖经营周报”或“新品试餐后整理反馈报告”。

输入清单

明确需要哪些文件、表格、链接、图片，缺少时先向人确认。

执行步骤

像教实习生一样写步骤，避免让模型自行猜流程。

判断规则

指标口径、阈值、品牌禁区、数据可信度都要写清楚。

输出模板

固定成网页、表格、报告、PPT 页、日报或待办清单。

验收方式

谁验收、检查什么、失败如何重跑、如何归档。

Agent 制作 · 启动营最小交付包

今天每组做的不是完整系统，而是一个“最小可行 Agent 包”只要这四件事齐了，就能从 demo 继续迭代成部门工作流。

01

一张 Spec

说明业务目标、输入资料、处理步骤、判断规则、输出格式和验收标准。

02

一份样例输入

真实表格、课件、菜单、评价、视频脚本或竞品资料，不能只有口头描述。

03

一个可打开产出

网页、报告、清单、表格、模板或脚本，现场能展示给同事看。

04

一条迭代计划

下周补什么数据、谁验收、进入哪个会议或日常动作。

上下文工程 · 像培养新人一样培养 Agent

多数 AI 结果不好，不是模型不行，而是上下文不够资料、规则和样例越完整，Agent 越接近一个可被管理的执行者。

资料

把真实材料给它

历史报表、调研数据、菜单、产品图、投放记录、门店反馈、竞品链接。

规则

把判断标准写明

什么算好内容、好点位、好产品、好回复、好报表，不能只说“高级一点”。

样例

把过往案例喂进去

成功案例、失败案例、标准课件、品牌话术、老板认可的输出样式。

规格驱动 · 把模糊需求写成工作合同

任何部门任务，先填这 8 格一张简单的 Spec 表，可以把“帮我做一下”变成可执行任务。

1. 业务目标为什么做，解决哪个经营问题。
2. 使用者谁每天/每周会用这个产出。
3. 输入资料需要哪些表格、文档、图片、链接。
4. 处理步骤像教实习生一样写清流程。
5. 判断规则哪些指标、阈值、口径要遵守。
6. 输出格式网页、表格、PPT、日报、清单、脚本。
7. 验收标准什么情况下算可用，谁来确认。
8. 复用方式如何沉淀为模板、Skill 或看板。

工具体系 · 日常协作与工程化交付分工

WorkBuddy 与 Codex 的分工：一个让大家上手，一个把流程做扎实轻量任务先跑通，复杂流程再进入可版本管理、可交付的工程化形态。

WorkBuddy

适合快速对话、资料整理、轻量生成、个人日常提效。今天让大家先跑通，降低上手门槛。

Codex

适合把复杂流程做成网页、脚本、模板、可版本管理的交付物。后续重点用于部门级 Agent 项目。

实操原则 · 小、真、可见、可验收、可复用

现场实操只遵守五个原则先做能跑的小闭环，再逐步扩展成部门级工作流。

真

真实业务

不用玩具题，必须来自部门访谈。

小

先做小闭环

下午只做第一版，不追求大系统。

显

能看见

网页、表格、报告、模板必须可展示。

留

能沉淀

输出要进入公司资料库，不能只在个人对话里。

场景 01 · 市场投放

市场不是多发内容，而是建立“投放-核销-复购”的增长模型把活动执行变成可衡量、可复盘、可加量的增长闭环。

痛点

投出去不等于赚到钱

GMV、核销率、毛利、复购、品牌调性没有被放在一个模型里看。

Agent

每周投放复盘助手

读取投放数据和核销数据，输出哪些加量、哪些暂停、哪些改套餐。

Demo

爆款内容模板研发

拆解 3 条竞品视频结构，生成适合品牌的脚本模板和拍摄清单。

场景 02 · 外卖运营

外卖增长的核心不是做图，而是发现商圈机会先看清商圈需求、竞品变化和价格带，再决定选品和营销动作。

痛点

机械工作多

多后台导数、评价回复、竞价调整、竞品观察占用大量时间。

Agent

外卖机会面板

定期抓取商圈竞品月销、价格带、品类变化、评价关键词。

Demo

人工导出 + AI 清洗

先不碰账号密码，先把导出的 Excel 自动清洗并生成部门周报。

场景 03 · 产品研发

新品研发要从“拍脑袋”，走向“测试-反馈-定价”AI 的作用是把分散信号整理成可讨论、可验证的产品决策依据。

痛点

灵感有，基准少

Ins、小红书、点评能提供灵感，但算法推荐会放大偏见。

Agent

新品调研助手

抓取关键词、竞品菜单、评价痛点，生成热度、差异化、价格区间报告。

Demo

试餐反馈分析

把随机用户试餐录音/表单整理成口味、价格、复购意愿分析。

场景 04 · 门店运营

运营要从月度事后复盘，走向每周异常预警能按周发现的问题，就不要等到月底才进入损益表。

痛点

数据晚，动作散

现场不追求大系统，只追求真实资料、清晰 Spec、可打开产出和下一周能继续迭代的负责人。

实操产出 · 每组一个可展示小闭环

每组只需要交付一个可展示的小闭环第一版必须足够小，确保现场能完成；同时必须来自真实业务，确保后续值得迭代。

输入

1 份真实材料

流程

1 张 Spec

输出

1 个 Demo

网页、报告、清单、表格、模板、脚本、看板雏形，现场能打开看。

任务 A · 市场内容模板

市场组：从一条爆款视频，拆成可复用内容模板从制作单条视频，升级为研发可复用的内容结构。

适合市场、视频、平面

是否能每周复用生成新脚本？

任务 B · 外卖机会面板

外卖组：把导出的多平台数据变成机会面板先用文件型自动化跑通流程，再评估是否进入账号登录自动化。

适合外卖、财务、运营

先采用安全模式：人工导出数据文件，AI 负责读取、清洗、汇总、解释。

输入美团/饿了么/收银导出的样表。
流程字段识别、口径统一、异常标注、趋势摘要。
输出HTML 周报或 Excel 汇总表。
能否减少每周重复整理？
能否指出该做什么动作？

任务 C · 新品反馈分析

研发组：把试餐反馈变成可决策的分析报告用户反馈只有被结构化，才能支持研发、定价和上市决策。

适合研发、市场、门店

围绕新品测试，把随机用户反馈整理成口味、价格、复购、顾虑四类结论。

输入试餐问卷、访谈录音转写、价格选项。
流程标签化、情绪分析、价格敏感、改进建议。
输出新品测试报告 + 定价建议页。
是否能降低上市前的不确定？
是否能给到下一轮测试问题？

任务 D · 门店经营诊断

运营组：做一份门店经营周报诊断让管理者先看到前三个异常，而不是被所有数字淹没。

适合运营督导、店长、财务

把营业、人力、毛利、差评放在一页里，输出本周前三问题和建议动作。

输入一周营业、人力、毛利、评价样表。
流程目标对比、异常识别、原因假设、动作建议。
输出红黄绿经营诊断页。
店长是否一眼看懂？
下周是否能复盘动作结果？

任务 E · 培训课件标准化

人事组：从旧课件到新模板，先跑通一页先用 HTML 模板统一版式，再把课件资产逐步批量迁移。

适合 HR、培训、营运

选择一页旧课件，让 AI 提取内容、套入新模板、生成可展示的培训页。

输入旧 PPT 截图/文字、新菜单表、模板样式。

适合采购、财务、供应链

拿两份样表，定义匹配字段、容差规则、异常类型，让 AI 生成差异清单。

输入采购单、收货单、平台结算表、补贴表。

规则是否可复用到下周？

门店开发组：做一个点位调研或 B 端招商页把分散的商圈、竞品和品牌信息，整理成可讨论的决策证据。

适合开发、加盟、市场

选一个商圈或一个加盟主题，把公开数据和品牌资料整理成可沟通材料。

是否适合持续更新？

验收标准 · 用同一把尺子看产出

现场 Demo 用同一把尺子验收一个 demo 的价值，不在于当场炫技，而在于下周能继续改进。

输入资料是真实业务资料，不是空想题。

管理层视角 · 从个人提效到组织能力

AI 资产管理的核心，不是保存文件本身，而是沉淀可复用的工作方法。

如果工作流长期停留在个人电脑、个人账号和聊天记录中，公司最终只能拿到结果文件，无法获得可移交、可复制、可持续优化的规则、样例和执行步骤。

资产定义 · AI 工作流不只保存最终版

AI 资产需要同时保存结果与生成过程相较普通文件归档，AI 工作流更需要保留输入、提示词、处理步骤、人工修订和复用条件。

01

Prompt、Skill、Workflow、SOP、脚本、配置和工作流说明卡。

02

数据样例资产

输入样例、业务规则、字段口径、成功案例、失败案例和参考输出。

03

业务成果资产

报告、脚本、周报、对账清单、调研页、课件模板和看板雏形。

04

治理记录资产

负责人、版本记录、人工修订、验收结果和后续迭代记录。

制度边界 · 先明确资产归属和使用责任

BYOD 与 AI 资产补充协议应作为制度前提在员工使用个人设备和 AI 工具的阶段，必须先明确资产归属、提交义务、敏感数据边界和离职移交要求。

明确公司资料、Prompt、Skill、Workflow、输出案例等属于公司可管理资产。
要求员工使用指定目录、公司账号、仓库和云文档空间沉淀工作产出。
涉及客户、员工、财务、供应链、账号和合同等敏感信息时，必须遵守脱敏和授权规则。
员工离职、调岗或项目结束时，应完成 AI 资产清单、权限和关键流程移交。
公司账号、GitHub、飞书、邮箱和 AI 工具权限，应具备统一回收机制。
对漏交、隐匿、外传或离职后违规使用的情形，应保留制度和法律追责依据。

轻量化测试 · 先确认是否有值得沉淀的真实资产

建议先跑一个月轻量化测试本次实地辅导之后，到下次进场刚好形成一个月观察周期，重点看真实流程是否开始流转。

观察

是否有真实流程流转

下次进场重点看部门是否真的把作业带回日常工作，并形成输入、处理、输出和复盘链路。

打磨

先跑通真实业务场景

如果没有有效流程，优先继续帮助客户把场景跑通，而不是提前投入资产库和自动化工具。

前提

资产沉淀建立在价值之上

资产沉淀是第一优先级任务，但前提是已经产生值得沉淀、复用和评审的 Prompt、Skill 或工作流。

一个月测试期的判断重点：先看有没有可复用资产，再决定进入哪一级资产管理方案。

版本建议 · 三档投入路径

本次建议提供三个版本，按成熟度循序推进不建议跳过手动规范期；先通过培训和制度建立资产意识，再根据数据质量决定是否进入自动化版本。

01

人工培训版

以培训、制度和模板驱动员工手动提交，适合作为第一个月正式规范期。必须同步说明：手动操作不可持续，容易漏传、格式不统一、说明卡缺失。

02

一键上传版

如果一个月后 GitHub 数据仍不整洁，或员工提交不稳定，就必须推进一键上传自动化，把目录、命名、说明卡、上传和日志标准化。

03

公司设备版

在版本二基础上，针对关键岗位配置公司电脑或专用设备，进一步提升本地数据、账号权限和离职交接的可控性。

建议路径：轻量测试一个月，版本一手动规范一个月；若数据质量仍无法稳定，即进入版本二。

版本二触发条件 · 手动规范不可持续时进入自动化

一键上传不是第一天就上，而是为了解决手动提交失控当 GitHub 数据质量、提交频率和说明卡完整度无法稳定时，就需要用工具固化资产沉淀动作。

触发条件

如果版本一运行一个月后，仍出现漏传、命名混乱、只交结果不交方法、说明卡缺失或负责人审核成本过高，即建议进入版本二。

价值

降低员工提交摩擦

自动归类文件，生成 daily-log.md、workflow-card.md、manifest.json，减少员工理解目录和命名规则的成本。

留痕

形成可检查的提交记录

自动 commit，记录成功/失败、提交时间、负责人和文件清单，方便部门负责人做周度检查。

边界

先解决方法论资产

第一阶段聚焦 GitHub 资产上传；飞书大文件、复杂权限和自动质量审核可作为后续升级项。

进度评估 · 与六个月咨询周期对齐

建议按月评估，逐步升级资产管理强度先用一个月观察真实流程，再用一个月规范手动提交，第二个月末评估是否进入版本二。

M1

轻量化测试

观察是否有真实流程流转；如果没有，先继续打磨业务场景。

M2

版本一：手动规范

通过培训、模板和制度要求员工手动提交 GitHub 与飞书资产。

M2 末

评估是否进入版本二

如果 GitHub 数据不整洁、说明卡缺失或提交不稳定，启动一键上传自动流。

M3-5

周度评审与资产库沉淀

持续筛选可复用工作流，沉淀为共享 Skill、模板和业务案例库。

M6

评估飞书与设备升级

咨询期结束时，评估飞书容量升级；部分关键岗位可配置专用电脑。

组织分工 · 技术、制度与管理同步落地

AI 资产治理需要管理层、部门负责人和员工共同承担工具只能降低整理和提交成本；真正形成组织资产，需要制度约束、负责人评审和持续使用。

管理层

确定制度与优先级

指定 AI 资产负责人；定期查看评审结果；推动 HR/法务完善协议；对优秀工作流设置激励。

部门负责人

审核与筛选资产

每周检查本部门提交情况，判断哪些流程进入共享库，并负责离职或调岗时的 AI 资产交接验收。

员工

按规范提交过程资产

将 AI 产出放入指定目录，说明业务用途、输入资料和处理规则，并按要求上传或一键提交。

目标结果：公司形成可继承的 AI 工作流资产，管理层看到真实落地进度，部门之间减少重复建设。

战略延展 · 从内部资产到行业级 Agent

数据资产沉淀得越扎实，未来越有机会形成餐饮行业 Agent这不是第一阶段要立刻建设的系统，而是建立在长期资产积累之上的产品化可能。

01

内部先形成轻量 Agent

把门店运营、外卖、市场、研发、供应链、财务等高频流程沉淀为可复用 Skill 和工作流。

02

再沉淀行业方法论

当案例、字段、模板、判断规则足够丰富后，可以抽象出适用于连锁餐饮企业的通用能力模块。

03

形成产品化和资本故事

未来可面向同行输出行业 Agent、SaaS 化工具或咨询产品，也可作为融资时的数字资产和数据壁垒。

关键前提之一：前期的数据资产、流程资产和案例资产要持续沉淀，并通过评审保证质量、可复用性和可追溯性。

管理层收束 · 数字员工团队

把数字员工团队
整理成公司的长期资产。

每一条 Prompt、每一个 Skill、每一套 Workflow，都是未来可复用、可训练、可扩展的组织能力。

01

最忠诚

数字员工不会离职，也不会把公司积累的经验带走。

02

可复制

一个部门跑通的工作流，可以复制到新门店、新品牌和新团队。

03

可进化

每一次复盘、修订和提交，都会让公司的 AI 能力继续变强。

04

带不走

员工会流动，但沉淀好的数字员工团队会留在公司。
